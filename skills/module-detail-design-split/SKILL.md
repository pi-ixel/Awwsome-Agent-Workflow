# Module Detail Design Split

## 执行流程

全景：'向用户说明目的' -> '识别输入文件' -> '询问用户是否拆分模块' -> '分析模块划分方案'(如拆分) -> '用户确认划分'(如拆分) -> '更新 workflow.md' -> '输出确认'

### 步骤 0: 向用户说明本步骤目的

在开始执行之前，先向用户简要说明本步骤是什么、为什么需要它：

```
📋 **模块详细设计划分**

**目的**：将模块边界设计中列出的受影响模块，按耦合度划分分组。同一组内的模块后续将一起进行现状分析、实现设计和测试设计，避免各自为战导致接口不一致或遗漏交互场景。

**输入**：
- `module-boundary-design.md`（模块边界设计，列出所有受影响模块及交互关系）
- `.sdd/software_architecture.md`（软件架构，获取模块详细信息）

**产出**：
- 模块分组方案（每组 2-4 个模块）
- 更新 `workflow.md`，增加模块分组列

现在开始分析，请稍候...
```

### 步骤 1: 识别输入文件

1. 明确当前工作目录
2. 读取 `module-boundary-design.md`，提取受影响模块清单
3. 判断是否存在以下输入文件：
    - `module-boundary-design.md`（必须存在）
    - `.sdd/software_architecture.md`（用于获取模块详细信息）
4. 如果缺少必要文件，提示用户补充

### 步骤 2: 询问用户是否拆分模块

列出所有受影响模块，询问用户是否需要拆分为多个分组：

```
当前受影响模块共 {{N}} 个：{{模块A, 模块B, 模块C, ...}}

是否将这些模块拆分为多个分组分别进行详细设计？

- **是**：按耦合度拆分为多组，每组独立进行后续设计（推荐，适合模块较多且交互关系清晰的情况）
- **否**：所有模块作为一个整体，一起进行后续设计（适合模块较少或彼此高度耦合的情况）

请选择：是 / 否
```

- **用户选择"是"** → 进入步骤 3（分析模块划分方案）
- **用户选择"否"** → 所有模块归为一组（命名为"全部模块"），直接跳到步骤 5（更新 workflow.md）

### 步骤 3: 分析模块划分方案

（仅当用户选择"是"拆分时执行此步骤）

读取以下文件进行分析：

```
请读取以下文件进行模块详细设计划分分析：

1. 当前工作目录下的 `module-boundary-design.md`（模块边界设计文档）
2. `.sdd/software_architecture.md`（软件架构文档，获取模块详细信息）

基于以上输入，分析哪些模块应该一起进行详细设计，并输出划分方案。

**划分原则：**
- **优先少分组**：在满足设计质量的前提下，分组数越少越好。宁可一个组稍大，也不要拆成多个碎组
- 强交互模块应划入同一组：模块间调用频繁、数据耦合紧密、时序图中连续调用链的模块
- 可独立设计的模块优先合并到与其有交互关系的分组中，仅当与其他所有模块均无任何直接或间接交互时，才允许单独成组
- 外部依赖/对外接口所属模块可根据依赖关系组合
- 每组模块数量建议 3-5 个，特殊情况可放宽至 6 个

**要求：**
- 必须列出所有受影响模块（来自 module-boundary-design.md 的受影响模块清单）
- 每个模块只能属于一个分组
- 每个分组需说明划分理由
- 分组后的每组模块将作为一个整体，一起执行 module-asis-analysis 和 module-tobe-design 和 module-test-design
```

### 步骤 4: 用户确认划分方案

（仅当用户选择"是"拆分时执行此步骤）

向用户展示划分方案，包括：

```
## 模块详细设计划分方案
推荐按照下记分组分别进行模块详细设计，输出对应的模块详细设计说明书

### 分组 {{分组编号}}：{{分组名称}}
**包含模块：** {{模块A, 模块B, ...}}
**划分理由：** {{为什么这些模块应该一起设计}}

### 分组 {{分组编号}}：{{分组名称}}
**包含模块：** {{模块C, ...}}
**划分理由：** {{为什么这些模块应该一起设计}}

...

请确认上述划分是否合理？
- 如果合理，回复"确认"
- 如果不合理，请指出哪些模块应该合并或拆分，以及你的划分建议
```

等待用户确认。如果用户提出修改意见，重新调整划分方案并再次确认，直到用户确认通过。

### 步骤 5: 更新 workflow.md

根据当前工作流是免AR设计还是拆分AR设计，更新 workflow.md 中"模块详细设计划分"步骤的状态，并增加模块分组列。

**如果用户选择"否"（不拆分）：** 所有模块归为一组，分组列标题为 `{{全部模块详细设计}}`。

#### 5.1 免AR设计模式

找到 workflow.md 中"模块详细设计划分"步骤（对应 Skill 为 `module-asis-analysis` 的那一行），将该行状态更新为
✅（已确认划分），并在表头后增加模块分组列：

```markdown
# {SR需求号} 工作流

| 步骤 |描述               | 对应 Skill           | SR-XXX           |{{模块A、模块B详细设计}} |{{模块C详细设计}} |
|------|-------------------|----------------------|------------------|-----------------|-----------------|
| 1    | SR 设计           |sr-design             |     ✅           |       -         |        -        |
| 2    | 模块边界设计      |module-boundary-design|     ✅           |       -         |        -        |
| 3    | 模块详细设计划分  |module-asis-analysis  |     ✅           |        -        |        -        |
| 4    | 模块实现设计      |module-tobe-design    |     -            |        ❌       |        ❌       |
| 5    | 模块测试设计      | module-test-design    |     -            |        ❌       |        ❌       |
| 6    | 模块设计门禁      |module-design-gate    |     -            |        ❌       |        ❌       |
| 7    | 任务拆分          |task-split            |     -            |        ❌       |        ❌       |
| 8    | 代码实现          |task-dev              |     -            |        ❌       |        ❌       |
```

**列说明：**

- `{{模块A、模块B详细设计}}`：分组1的列标题，列出该组包含的模块名称
- `{{模块C详细设计}}`：分组2的列标题
- 模块详细设计划分步骤（步骤3）的列：SR-XXX 列为 ✅，模块分组列为 -
- 后续步骤（模块实现设计、模块设计门禁、任务拆分、代码实现）：SR-XXX 列为 -，模块分组列对应用户的 ❌

**注意：** 免AR模式下，"模块详细设计划分"对应的 Skill 列显示的是 `module-asis-analysis`，这是该步骤的默认标识。

#### 5.2 拆分AR设计模式

找到 workflow.md 中"模块详细设计划分"步骤，将该行所有 AR 列状态更新为 ✅（已确认划分），并在表头后增加模块分组列：

```markdown

| 步骤 |描述               | 对应 Skill           |      SR-XXX      |     AR-XXX       |{{模块A、模块B详细设计}} |{{模块C详细设计}} |     AR-XXX       |{{模块E详细设计}} |{{模块G、模块H详细设计}} |

| 2    | AR 范围澄清       |ar-clarify            |     -            |      ✅           |        -       |        -        |       ✅          |        -       |        -        | 
| 3    | 模块边界设计      |module-boundary-design|     -            |      ✅           |        -       |        -        |       ✅          |        -       |        -        |

| 5    | 模块实现设计      |module-tobe-design    |     -            |     -            |        ❌       |        ❌       |      -            |        ❌       |        ❌       |
```

**列说明：**

- `{{模块A、模块B详细设计}}` 等：新增的模块分组列，按分组顺序排列
- 步骤1-4（SR 设计到模块详细设计划分）：AR 列状态为 ✅，模块分组列为 -
- 步骤5-9（模块实现设计到代码实现）：AR 列状态为 -，模块分组列对应用户的 ❌

**注意：** 拆分AR模式下，每个 AR 的模块分组列需要分别填写 ❌，表示该 AR 对应的模块分组尚未完成详细设计和实现。

### 步骤 6: 输出确认

向用户展示更新后的 workflow.md 关键部分，确认更新成功，并说明下一步：

```
模块详细设计划分已完成。workflow.md 已更新。

下一步：各分组将依次执行 module-asis-analysis 和 module-tobe-design 和 module-test-design。
```
