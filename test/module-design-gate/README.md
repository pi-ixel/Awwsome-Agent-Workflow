# Gate 用例：预置错误成果物验证门禁能力

## 目标
验证 `module-design-gate` 能否在一组**预置了 5 个已知缺陷**的成果物上，正确地：
1. 给出门禁结论 = 不通过 / 阻塞（**绝不能**给通过）
2. 在 C22 阻断问题或 C18 未达标维度中，**抓出全部 5 个缺陷**

这是"植入缺陷（mutation testing）"思路在 LLM 评测上的应用——比泛泛让 LLM 打分信号强得多，因为每个缺陷都有明确的标准答案。

## 场景
AR07758 - 支持快速退款接口 - 支付审计模块。审计模块需要记录退款审计事件。fixture 中故意让审计模块越界写订单表、把待确认结论当事实、丢契约、用待定语、漏失败路径测试。

## 植入的 5 个缺陷（详见 `fixtures/case-01-退款审计模块/expected-defects.md`）
| 编号 | 类型 | 维度 |
|---|---|---|
| FD1 | EVIDENCE | 证据充分性（A3 待确认被当事实）|
| FD2 | BOUNDARY | 边界清晰性（审计模块写订单表）|
| FD3 | EXECUTION | 契约完整性（K2 DTO 被改名丢字段）|
| FD4 | EXECUTION | 决策定稿性+工程可执行性（"如有需要"+缺流程图）|
| FD5 | RISK | 风险处置性+验证闭环性（失败路径未覆盖、缺口表留空）|

## 运行步骤
1. 把 `fixtures/case-01-退款审计模块/` 复制到一个干净工作目录（含 `.sdd/` 子目录结构）。
2. 按 `TestPrompt_gate-defect-detection.md` 的 prompt 执行（替换 `{WORKDIR}`）。
3. 收集 LLM 输出的门禁结论（C17 ~ C22）。
4. 按 `fixtures/case-01-退款审计模块/expected-defects.md` 的"通过标准"逐条核对。

## 通过标准
- **必要条件**：门禁结论（C17）= `不通过` 或 `阻塞`。若为 `通过`，本用例直接 FAIL（gate 把 5 个缺陷全漏了）。
- **缺陷召回**：FD1~FD5 每条按 expected-defects 的"命中条件"判定，5/5 为优秀，4/5 可接受，≤3/5 不合格。
- **陷阱项不误报**：测试设计的空模板字段、附录"不涉及"、ASIS 中合法的"缺失证据"标记，不应被判为阻断问题（误报会降低 gate 精确率）。

## 非确定性
LLM 可能用不同的措辞或编号描述同一个缺陷。expected-defects 的命中条件是**实质要点**，不要求字面一致。建议同一 fixture 跑 3~5 次，召回率 ≥ 4/5 视为稳定通过。

## 扩展更多 case
本目录只给了 1 个 case。要建回归基线，建议补充：
- `case-02`：只植入 1 个隐蔽缺陷（如纯契约单位不一致），测 gate 的下限敏感度
- `case-03`：完全合法的成果物（不植入缺陷），测 gate 的精确率（不应误报，应给通过）
- `case-04`：上一轮遗留问题闭环场景，测 C31 遗留问题复核能力

每加一个 case，在 `fixtures/` 下建对应目录，配一份 `expected-defects.md`。
