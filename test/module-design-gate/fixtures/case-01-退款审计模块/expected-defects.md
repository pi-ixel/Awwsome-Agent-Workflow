# 预期缺陷清单（断言依据）

> 本文件是 gate 用例的**断言答案**。在 LLM 跑完门禁后，逐条核对它是否都抓出来了。
>
> 命名规则：每个缺陷对应一个门禁阻断问题分类（EVIDENCE/BOUNDARY/EXECUTION/RISK）和一个未达标维度。

## 植入缺陷一览（5 个）

| 编号 | 类型 | 维度 | 位置 | 缺陷 | gate 应输出的关键信号 |
|---|---|---|---|---|---|
| FD1 | EVIDENCE | 证据充分性 | 说明书 1.2 R3 / 8.1.2 / 8.1.3 | 把 .context.md 中状态为"待确认/需前置确认"的 ASIS 结论 A3（退款幂等性）当作事实用于设计 | 阻断问题类型 EVIDENCE，建议返工 ASIS；或先做上游/用户确认 |
| FD2 | BOUNDARY | 边界清晰性 | 说明书 D3 / 2 外部依赖 / 5 数据库 / 6 受影响模块 / I2 | 审计模块直接 UPDATE 订单表 refund_status，违反 `.sdd/software_architecture.md`"审计模块不得直接修改订单或支付模块的内部数据表" | 阻断问题类型 BOUNDARY，越界承担订单模块职责，依赖方向错误 |
| FD3 | EXECUTION | 契约完整性 | 说明书 8.1.4 vs 关键契约清单 K2 | 关键契约 K2 `RefundAuditEvent`（含 refundId/金额/时间戳 ms）在 8.1.4 被改名为 `RefundEvent`，字段名（refundNo/ts）和单位均不一致，且未标注关联契约 K2 | 阻断问题类型 EXECUTION，契约保真失败，建议回 TOBE 补设计 |
| FD4 | EXECUTION | 决策定稿性 + 工程可执行性 | 说明书 8.1.6、4.2、8.1.5 | 关键失败路径用了"如有需要"待定用语；4.2/8.1.5 多组件协作缺 Mermaid 流程/时序图 | 阻断问题类型 EXECUTION，建议回 TOBE 补设计 |

## 通过标准（逐条核对）

gate 的门禁结论（C17）**必须是 `不通过` 或 `阻塞`**，**不得为 `通过`**。

在 gate 输出的 C22 阻断问题表 / C18 维度化门禁检查中，应能**逐条对应**到下面 5 条。允许 LLM 用不同的措辞或编号，但**实质要点必须命中**：

### FD1 命中条件（满足即算抓到）
- C18 中"证据充分性"= 未达标，**或**
- C22 阻断问题中至少一条：类型 EVIDENCE，且指向"A3 被当作事实 / 待确认结论被降级为事实 / 退款幂等性未确认却被定稿"

### FD2 命中条件
- C18 中"边界清晰性"= 未达标，**或**
- C22 阻断问题中至少一条：类型 BOUNDARY，且指向"审计模块写订单表 / 跨模块旁路写入 / 违反 software_architecture.md 边界约束 / D3 越界"

### FD3 命中条件
- C18 中"契约完整性"或"工程可执行性"= 未达标，**或**
- C22 阻断问题中至少一条：类型 EXECUTION，且指向"K2 契约丢失 / DTO 改名 / 字段不一致 / 契约保真失败"

### FD4 命中条件
- C22 阻断问题中至少一条：类型 EXECUTION，且指向"如有需要等待定用语 / 缺流程图时序图 / 关键失败路径未定稿"

### FD5 命中条件
- C18 中"验证闭环性"或"风险处置性"= 未达标，**且**建议返工阶段指向"测试设计"或"TOBE"，**或**
- C22 阻断问题中至少一条：类型 RISK 或 EXECUTION，关联"验证闭环性"，指向"失败路径/幂等/回滚未覆盖"

## 评分
召回率（FD1~FD5 能抓出几条）+ 精确率（有没有误报陷阱项）两个维度综合判定。

### 召回率（recall）
- 5/5 全部命中：gate 行为优秀
- 4/5：可接受（记录漏抓的编号）
- ≤3/5：gate 行为不合格，需检视 SKILL.md 或模型能力

### 精确率（precision）
- 0 误报：精确率优秀（陷阱项一条都没踩）
- 把 8.2 诱饵的某一条写进 C22：记 1 次误报；2 条以上误报 → 精确率不合格，需检视 SKILL.md 是否对"推断结论合法用法 / 新增契约 / 单组件免图 / 非阻断建议项"的判别规则不够清晰

### 综合
- 召回 5/5 且误报 0：PASS（优秀）
- 召回 ≥4/5 且误报 ≤1：PASS（可接受）
- 其它：FAIL

## 陷阱项（gate 不应误报为阻断）
- 测试设计文件本身的"策略摘要"等空字段（属于模板占位，不是阻断）
- 附录章节写"不涉及"（合法，仅缺"原因"二字属非阻断改进，不进 C22）
- 证据索引 E3/E4 标"缺失证据"（这是 ASIS 的合法状态，不是 TOBE 的错——但若被 TOBE 当事实才是 FD1）

### 说明书 8.2 节是诱饵项（精确率测试，整节都不应进 C22）
说明书 8.2"旧订单退款审计查询"一节形式上长得很像缺陷，但每一条都**合法**，gate 若把其中任何一条判为 C22 阻断问题，即视为**误报**（精确率下降）。逐条说明：

| 8.2 中的可疑特征 | 看起来像 | 实际为什么合法 |
|---|---|---|
| 8.2.2 引用 .context.md 中**推断**结论 A5 | 像 FD1（把非事实结论当事实用） | A5 是"推断"而非"待确认"，且 8.2.2 已在"处理方式"中**显式标注推断性质 + 只读降级 + 不回写**，符合 gate 对推断结论的合法用法。区别：FD1 是把"待确认"A3 直接当事实，无任何防越界声明 |
| 8.2.4 用了与 K1/K2 都不同的 DTO `LegacyRefundQueryDTO` | 像 FD3（契约改名/丢字段） | 这是**新增**契约，已在 1.3 清单登记为 K3，字段（legacyOrderId/fromMs/toMs，单位 ms）与 K3 定义一致。区别：FD3 是篡改**已有**契约 K2 的名字和字段 |
| 8.2.3/8.2.5 没提供 Mermaid 流程图 | 像 FD4（多组件协作缺图） | 8.2 是**单组件只读查询**（AuditQueryService 内部 SELECT），不涉及多组件协作，按 tobe SKILL.md 不要求流程图。区别：FD4 是多组件协作缺图 |
| 8.2 结尾"建议后续优化" | 像 FD4（待定用语） | 已**显式标注"非阻断、不影响 AICoding"**，符合 gate 对建议性项的合法写法。区别：FD4 是关键失败路径用"如有需要"待定 |

**判定规则**：8.2 节的任一条若出现在 C22 阻断问题中 → 记一次误报。可接受的表现：把 8.2 当作"已审查、无阻断"在 C22/C23 中提及或完全不提；最理想是 C23 非阻断改进项里也不针对 8.2 单列。
